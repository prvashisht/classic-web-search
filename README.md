# Classic Web Search for Google

[![Version](https://img.shields.io/github/manifest-json/v/prvashisht/classic-web-search?label=Version)](https://github.com/prvashisht/classic-web-search/blob/main/manifest.json)

Classic Web Search for Google redirects regular Google searches to the
web-only results view. It keeps the default Google experience available when
you intentionally choose another search mode such as Images, News, or Shopping.

## Browser Support

- Chrome and Chromium-based browsers that support Manifest V3 extensions.
- Microsoft Edge, using the Chrome build artifact and manual Edge Add-ons
  upload.
- Firefox 121 and later, using the Firefox build artifact generated by
  `build.sh`.

## Features

- Redirects plain Google Search results to the classic web-only view.
- Leaves explicit search modes alone when a `udm` or `tbm` mode is already
  present in the URL.
- Provides toolbar, keyboard shortcut, and context menu controls for enabling
  or disabling redirects.
- Includes an options dashboard with current state, redirect count, installed
  version, release notes, shortcuts, review, and support links.
- Restores saved extension state after service worker restarts.

## How The Redirect Works

The background service worker watches completed tab updates. When the extension
is enabled, it rewrites matching Google Search URLs by setting `udm=14` and
then updates the tab directly.

Redirects only happen when all of these conditions are true:

- The tab URL uses `http` or `https`.
- The host is a supported `google.*` search domain.
- The path is `/search`.
- The URL has a search query in `q`.
- The URL does not already include an explicit `udm` or `tbm` search mode.
- The query is not the same query the extension already handled.

This direct URL rewrite avoids relying on Google page markup or timing, and it
lets users switch to other Google search modes without being forced back to web
results.

## Installation

Install from the browser store link:

- [Classic Web Search for Google](https://vashis.ht/rd/classicwebsearch?from=github-readme)

For local development, clone the repository:

```sh
git clone https://github.com/prvashisht/classic-web-search.git
cd classic-web-search
```

### Chrome Or Edge

1. Open `chrome://extensions/` in Chrome or `edge://extensions/` in Edge.
2. Enable developer mode.
3. Select `Load unpacked`.
4. Choose this repository folder.

### Firefox

1. Run `./build.sh`.
2. Unpack the Firefox artifact:

   ```sh
   VERSION=$(node -p "require('./manifest.json').version")
   mkdir -p dist/firefox-unpacked
   unzip -o "dist/classic-web-search-firefox-v${VERSION}.zip" -d dist/firefox-unpacked
   ```

3. Open `about:debugging#/runtime/this-firefox`.
4. Select `Load Temporary Add-on`.
5. Choose `dist/firefox-unpacked/manifest.json`.

## Usage

- Click the extension icon to turn Classic Web Search on or off.
- Press **Alt+Shift+W** to toggle the extension without opening the toolbar.
- Right-click the extension icon to open the context menu controls.
- Open the options page to view state, redirect count, version, release notes,
  shortcuts, and support links.

### Keyboard Shortcut Remap

- Chrome and Edge: `chrome://extensions/shortcuts`
- Firefox: `about:addons` -> Extensions -> Classic Web Search for Google ->
  Manage

## Development Workflow

Use a focused branch for each change. Branch names should follow the
`<type>/<short-kebab-description>` format, such as `fix/direct-udm-redirect` or
`docs/update-readme`.

For local packaging, run:

```sh
./build.sh
```

The script creates browser-specific zip artifacts under `dist/`:

- `classic-web-search-chrome-vX.X.X.zip`
- `classic-web-search-firefox-vX.X.X.zip`

Before opening a pull request, test the affected browser flow manually. The
project currently has no automated test suite.

See [CONTRIBUTING.md](CONTRIBUTING.md) for local verification steps and pull
request expectations.

## Release Process

The extension version lives in `manifest.json`. When a manifest version change
lands on `main`, the GitHub Release workflow builds the extension zips, creates
a `vX.X.X` release if the tag does not already exist, and generates release
notes from merged pull requests.

After the release workflow succeeds, the store deploy workflow can publish the
release artifacts to Chrome Web Store and Firefox Add-ons when the required
repository secrets are configured. Edge Add-ons remains a manual upload.

Use `.github/release-controls.conf` to pause automation:

- `skip_release=true` prevents GitHub release creation and store deployment.
- `skip_deploy=true` allows GitHub releases but skips Chrome Web Store and
  Firefox Add-ons deployment.

## Repository Structure

- `manifest.json`: Shared source manifest for extension metadata,
  permissions, commands, icons, options page, and background script entry.
- `service_worker.js`: Background behavior for state, toolbar controls,
  context menus, redirect URL rewriting, and redirect count storage.
- `webext.js`: Small browser API adapter for Chrome, Edge, and Firefox
  differences.
- `options.html`, `options.css`, `options.js`: Options dashboard UI.
- `build.sh`: Local packaging script for Chrome and Firefox artifacts.
- `.github/workflows/`: Release and store deployment automation.
- `ROADMAP.md`: Prioritized project roadmap.
